# 2.1.2 - What is Kestra?

What is Kestra? Now Kestre is an open-source infinitely scalable orchestration platform that enables all types of engineers to manage business critical workflows. Now why Kestra is a great choice for workflow orchestration is because you can build your workflows as code, no code or with AI. And you can do all three of these things interchangeably. You can start by building your workflow in flow code. You can then transition to editing it with the no code editor. And then if you're not sure about adding an extra bit of functionality at the end, you can go and ask the AI co-pilot to add it for you. Ker also has a humongous plug-in library. When I say humongous, I mean over a thousand dedicated plugins, which means you can connect Kestra to all of your favorite tools. In this case, we're going to be connecting to things like Postgress, Google Cloud, as well as Python. And on the topic of Python, Kestra is actually language agnostic. Some orchestration platforms are only able to run Python, which is a great choice for a lot of use cases, but doesn't necessarily mean it's always the right tool for the job. Being language agnostic means if you need performance, you could use a language like Go or C. Whereas, if you need something that's maybe quicker to be able to write something together, then Python might be a good choice. And so having that option to pick a language that's good for the job rather than the one that your orchestrator uses gives you a lot of options. And on top of all of this, Kestra gives you full visibility into your workflows. With detailed logging, you can easily see what's going on in your workflows. You can set up error alerting, too, to notify you when things do go wrong, so you can jump in and fix them when needed. And lastly, an orchestrator wouldn't be an orchestrator without the ability to manage how your executions of your workflows start. Kesha has support for both schedule and event-based triggers, which means you have the option to decide how these workflows start, whether that's a nightly job running overnight or you have it set up where data arrives in your data lake or storage bucket and you can automatically start your workflows based on that. # 2.4.1 - ETL vs ELT

Welcome back to the next part of the data engineering zoom camp. Now, so far what we've covered is building our first data engineering pipeline using Postgress and we've also been able to add a schedule to this and run back fields to fetch data from past schedules. Now, this data pipeline has been using ETL also known as extract, transform and load. This is where we extract the data from our source in this case a CSV file stored on the web. Then we transform it. In our scenario, we've been adding unique ids and the file name to our data before we then merge it into our final destination where we combine all the data together in the load step. Now, this is standard across the industry where pipelines that run, , on a nightly schedule will run in this sort of format because it works very very well. However, sometimes there are cases where it may make sense to do things in a different order. Let's have a look at ELT. Now ELT stands for extract, load, transform. So instead of transforming your data before you load it into your final destination, you actually load it into the destination first and then transform on top of that data. Now the benefit here is this enables you to access things like the cloud where you can load your massive data set into the cloud and then you can start transforming it with the power of the cloud to help you. Now, if any of you tried running the yellow data set using your Postgress setup, you probably found it was incredibly slow because trying to add unique ids to millions of rows can take a beating, especially on a personal laptop. So, this is where we're going to start having a look at how we can use BigQuery in the next few episodes to allow us to interact with that yellow data set without it being such a hassle. So, let's break it down. What does ELT really look like in this scenario? So, we're going to start off by extracting our data just like we did from GitHub. We're going to make a HTTP request to get that CSV file. Now, what we're going to do is load it into a Google Cloud Storage bucket, which in this case, we're going to use as a data lake. Now, a data lake is where we can store our data. It doesn't need to be organized or sorted like it would in a database. So, in this case, we can just put the CSV file straight in our data lake and we can come back to it later. Now, if you'd love to learn more about data links, we recently just did a video on it. So, I'd recommend checking that out here. Now that we have our data in the cloud in our data lake, what we can do is now use our data warehouse, which in this case is BigQuery to be able to map to that data lake, use that CSV file in it, and now it can add some metadata on top. This means the data is stored in the data lake, but BigQuery is able to then utilize that data to perform our transformations and queries as we wish. The reason this is faster is not only because we're utilizing the cloud, which obviously is typically faster than a computer that you might have at home, but it isn't loading all of the data into BigQuery. Instead, it's just referencing it from the data lake, which one helps to keep costs a little bit lower, but two, it allows you to be able to perform big transformations without it getting super slow. This means the original data stays untouched and you're just doing queries on top of it. So you get the benefit of cheap storage, in this case a data lake rather than a database, but you get the power of the cloud to be able to get some insight into your data. Now in the next episode we're going to look at how we can modify our workflow that we already have to now use ELT and include Google Cloud Storage as our data lake and Big Query as our data warehouse.
